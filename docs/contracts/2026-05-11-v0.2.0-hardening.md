# Technical Contract: v0.2.0 Architectural Hardening

- **Requester:** Orchestrator
- **Status:** Pending Implementation
- **Version:** 0.2.0

## 1. Overview

This contract defines the architectural refactoring of the FastAPI backend to improve maintainability, testability, and adherence to FastAPI best practices. The focus is on implementing Dependency Injection (DI) for database sessions and modularizing the administrative API.

## 2. Dependency Injection Refactoring

### 2.1 Database Sessions

- **Current State:** Manual context managers `with Session(engine) as session:` are used across `main.py` and `admin.py`.
- **Target State:** All route handlers requiring database access must use `session: Session = Depends(get_session)`.
- **Scope:**
  - `backend/main.py`: All public CMS and analytics endpoints.
  - `backend/admin.py`: All CRUD operations.
  - `backend/seed.py`: Should also be updated to use a session provided externally where applicable.

### 2.2 Background Tasks

- **Target State:** Background tasks should also be refactored to use sessions properly. Since `Depends` only works in route handlers, the route handler should pass the session to the background task if possible, or the background task should manage its own session lifecycle using `get_session` as a context manager if appropriate.

## 3. Router Modularization

### 3.1 Structure

The current `backend/admin.py` monolith (11kb+) will be split into a modular structure under `backend/routers/admin/`.

- **Root Admin Router:** `backend/routers/admin/router.py` (Main entry point for admin routes).
- **Specialized Modules:**
  - `backend/routers/admin/about.py`
  - `backend/routers/admin/experience.py`
  - `backend/routers/admin/projects.py`
  - `backend/routers/admin/blog.py`
  - `backend/routers/admin/chat.py` (Handles triggers and feedback analytics)
  - `backend/routers/admin/uploads.py` (General upload logic)

### 3.2 Main API Consolidation

- Public routes currently in `main.py` should be moved to dedicated routers (e.g., `backend/routers/public.py`).
- `main.py` should serve primarily as the application factory and router aggregator.

## 4. Verification Plan

- **Unit Tests:** Run `pytest` to ensure no regressions in API functionality.
- **Admin Verification:** Manual check of `/admin` dashboard functionality (Projects, Blog, etc.).
- **NLU Check:** Verify chatbot still responds correctly to triggers.
- **Linting:** Ensure `ruff` passes with new structure.
