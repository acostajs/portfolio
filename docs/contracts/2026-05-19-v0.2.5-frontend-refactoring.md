# Technical Contract: V0.2.5 Frontend Refactoring & Hook Encapsulation

- **Requester:** Orchestrator
- **Status:** Completed (2026-05-19)
- **Version:** 0.2.5

## Scope

### 1. Custom Hook: `useLiveChat`
- **Goal**: Extract chat orchestration from `Home.tsx` into a reusable, testable hook.
- **Responsibilities**:
    - Manage message history state (with `localStorage` persistence).
    - Handle message sending (AI and Live Chat).
    - Manage loading and "isLive" states.
    - Implement the polling sync logic (temporary, until SSE in Phase 3).
    - Handle slash commands (`/clear`, `/help`, etc.).
    - Manage contextual hints and suggestions.

### 2. Component Decomposition
- **Goal**: Break down `Home.tsx` into atomic, focused components.
- **New Components**:
    - `ChatWindow.tsx`: Responsible for rendering the message list and loading state.
    - `ChatInputContainer.tsx`: Responsible for the input form, suggestions, and hints chips.
    - `Home.tsx`: Will act as the thin container using `useLiveChat` and passing props to the sub-components.

### 3. API Type Safety
- **Goal**: Eliminate `unknown` assertions in the API layer.
- **Action**: Refactor `promiseCache` in `frontend/lib/api.ts` to be type-safe or handle casting more gracefully within the `useSuspenseFetch` hook.
- **Reason**: Improves developer experience and reduces runtime errors.

## Acceptance Criteria
- [ ] `Home.tsx` is reduced to < 100 lines of code.
- [ ] `useLiveChat` hook encapsulates all chat logic and state.
- [ ] Chat functionality (commands, polling, persistence) remains identical to the user.
- [ ] `ChatWindow` and `ChatInputContainer` are modular and receive data via props.
- [ ] `useSuspenseFetch` no longer uses `unknown` in a way that requires unsafe casting.
- [ ] No regressions in chat UX or mobile responsiveness.
