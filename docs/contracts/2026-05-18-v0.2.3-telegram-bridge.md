# Technical Contract: Telegram Chat Bridge (v0.2.3)

## Status

- **Date**: 2026-05-18
- **Version**: 0.2.3
- **Status**: DRAFT

## 1. Goal

Implement a real-time (polling-based) bidirectional bridge between the portfolio's chatbot UI and a Telegram bot. This allows the developer to respond directly to portfolio visitors.

## 2. Backend Architecture

### 2.1 Configuration

- Update `backend/config.py`:
  - `TELEGRAM_BOT_TOKEN: Optional[str]`
  - `TELEGRAM_CHAT_ID: Optional[str]`

### 2.2 Models (`backend/models.py`)

- **ChatMessage**: Add `session_id: str` (index=True).
- **LiveChatSession**:
  - `id`: Primary Key.
  - `session_id`: Unique identifier from the client.
  - `is_active`: Boolean.
  - `created_at`: Datetime.
  - `updated_at`: Datetime.

### 2.3 API Endpoints (`backend/routers/chat.py`)

- **`POST /api/v1/chat`**:
  - Update `ChatRequest` to include `session_id`.
  - Check if `/live-chat` or `/close-live-chat` command is issued.
  - If a session is active for this `session_id`, relay message to Telegram via `BackgroundTasks`.
- **`POST /api/v1/chat/telegram-webhook`**:
  - Receive messages from Telegram.
  - Identify which session to relay back to (initially, we'll support one active session or use a mapping).
  - Save developer's reply as a `ChatMessage` with `role="assistant"` and the correct `session_id`.
- **`GET /api/v1/chat/sync/{session_id}`**:
  - Returns new messages for the given session since a specific timestamp.

## 3. Frontend Architecture

### 3.1 Session Management

- Generate a UUID `session_id` and store it in `localStorage` on first load.

### 3.2 Chat UI

- Intercept `/live-chat` and `/close-live-chat` commands.
- When `/live-chat` is active:
  - Change the UI state to indicate "Live Mode".
  - Start an interval (e.g., every 3-5s) to call `/api/v1/chat/sync/{session_id}`.
  - Stop polling on `/close-live-chat` or when the user leaves the page.

## 4. Verification Plan
