# Technical Contract: v0.1.9 Backend NLU Optimization

**Status:** Draft
**Owner:** Orchestrator
**Specialist:** backend-agent

## 1. Context & Vision

The current chatbot matching logic fetches all triggers from the database for every request and iterates through them using regex and fuzzy matching. As the trigger database grows, this will become a bottleneck. We need a two-tier matching system that prioritizes speed and accuracy.

## 2. Technical Specifications

### 2.1 Trigger Caching

- **Requirement:** Avoid redundant database queries for triggers on every chat request.
- **Action:**
  - Implement a simple in-memory cache for `ChatTriggerResponse` records.
  - Refresh the cache on startup and provide a mechanism to invalidate it (e.g., when an admin updates triggers).
  - Build a "Flat Trigger Map": A dictionary where keys are individual trigger strings (lowercase) and values are references to the parent `ChatTriggerResponse` object.

### 2.2 Two-Tier Matching System

- **Tier 1: Exact Hash-Map Lookup**
  - **Action:** Before running regex or fuzzy matching, tokenize the normalized user message and check for exact matches in the Flat Trigger Map.
  - **Note:** This handles single-word triggers extremely fast. For multi-word triggers, check if the trigger string exists as a substring of the normalized message.
- **Tier 2: Fuzzy Matching fallback**
  - **Action:** Only if Tier 1 yields no results, proceed to the current `rapidfuzz` logic.
  - **Optimization:** Use the cached triggers instead of re-fetching from DB.

### 2.3 Refactoring `is_trigger_match`

- **Action:**
  - Split the logic into `find_exact_match(user_message, cached_triggers)` and `find_fuzzy_match(user_message, cached_triggers)`.
  - Ensure priority is respected (the order in the database determines priority if multiple triggers match).

## 3. Acceptance Criteria

- [ ] Chatbot responses remain accurate and respect trigger priorities.
- [ ] Database query count for `/api/v1/chat` is reduced (ideally to 1 or 0 for triggers).
- [ ] Performance improvement for common exact-match queries.
- [ ] Admin updates to triggers are reflected in the chatbot (cache invalidation).

## 4. Constraints

- Maintain the current `rapidfuzz` threshold (85).
- Ensure multi-word triggers (e.g., "contact info") are still handled correctly in Tier 1.
